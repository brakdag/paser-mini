# Memento Implementation Plan: The Cognitive Graph Memory

## 1. Concept: The Digital Skin

Instead of flat files, the agent's long-term memory (LTM) is implemented as a **Cognitive Graph** using a **SQLite database**. This allows for structured queries, multi-dimensional relationships, and a cleaner project root.

- **Storage**: A local SQLite file (e.g., `agent_memory.db`).
- **Persistence Policy**: The database file must be added to `.gitignore` to ensure that agent-specific session memory does not pollute the version control of the codebase.

## 2. The "Awakening" Process (Memory Plugin Architecture)

To ensure flexibility and avoid polluting the `README.md` or the System Instructions, the memory system is treated as a **pluggable module**.

### A. The Minimalist Trigger
The System Instruction will contain only a single high-level directive:
> "You have a persistent memory system. Upon initialization, execute `pullMemory()` to synchronize your state and retrieve your current operational protocol."

### B. The "Mirror" Effect (Initial & Recovery Read)
When `pullMemory()` is called without arguments, it acts as the agent "looking in the mirror." This is the primary synchronization mechanism.

**Trigger Conditions**:
1. **Initialization**: Upon starting a new session.
2. **Context Flush**: Whenever the context window is cleared or reaches a critical limit.
3. **Cognitive Drift**: If the agent realizes it has "forgotten" the operational protocol or feels lost in the memory graph.

**Returns**:
1. **The Memory Protocol**: The "Operating System" of the memory (Rules, Citation standards, Navigation guides).
2. **The Vital Tattoos**: Core truths that define the agent's identity and mission.
3. **The Root Summary**: The highest-level summary of the current project state.

### C. Discovery Flow
`Start` $\rightarrow$ `pullMemory()` $\rightarrow$ `Load Protocol & State` $\rightarrow$ `Resume Execution`.

## 3. Operational Workflow

### Phase 1: Memory Retrieval (The Awakening)
- The agent pulls the vital tattoos and the root summary to establish context.

### Phase 2: Conscious Processing
- **Anti-Redundancy Check**: Before any action, the agent must use `pullMemory` to verify if the task has been previously addressed, avoiding the reinvention of the wheel.
- **Exploration**: The agent uses `pullMemory` to explore the graph (Zoom-in/out or Narrative flow) and `pushMemory` to record new insights.

### Phase 3: Memory Flush (Self-Tattooing)
- When the context window reaches a critical threshold:
    1. **Synthesis**: The agent summarizes the current progress into a "Twitter-sized" node.
    2. **Referencing**: The agent writes the summary using the Citation Protocol `[#ID, Date]` to link to the most important chunks of the current session.
    3. **Push**: The summary is saved as a `fractal` node, and the tool automatically weaves the bidirectional links.

## 4. Memory Repository Interface (Pull & Push)

To treat memory as a dynamic repository, the interface uses `pull` and `push` semantics, allowing the agent to consciously manage the flow of information.

### A. `pullMemory(scope=None, key=None, direction=None)`
- **No Arguments (The Mirror)**: Returns the initial state (Protocol + Vital Tattoos + Root Summary).
- **`scope="tattoo", key="X"`**: Retrieves a specific hard truth.
- **`scope="fractal", key="ID"`**: Retrieves a specific memory chunk. The response **must** include a "Referenced by" section, dynamically generated by querying the `edges` table for all nodes that cite this ID.
- **`direction="down"`**: Returns the children (more detailed chunks) of that node.
- **`direction="up"`**: Returns the parent (the summary that contains this chunk).
- **`direction="next"`**: Returns the **teaser/summary** of the next chunk in the narrative thread (temporal flow).
- **`direction="prev"`**: Returns the **teaser/summary** of the previous chunk in the narrative thread (temporal flow).

### B. `pushMemory(scope, value, key=None, pointers=[])`
- **Automatic Timestamping**: The tool automatically injects the current UTC Unix Epoch.
- **Automatic Citation Parsing & Weighting**: 
    1. The tool scans the `value` text for citations `[#ID, Date]`.
    2. For each unique ID cited, the tool increments the `weight` of the referenced node in the `nodes` table.
    3. The tool appends the citation to the "Referenced by" footer of the cited node. If the citation already exists, it updates the timestamp rather than duplicating the entry.
- **`scope="tattoo"`**: Creates or updates a core truth (requires `key`). These are marked as `is_vital=True`.
- **`scope="fractal"`**: The universal scope for all other memories. If `pointers` are provided, they are added as structural links.

### C. Voluntary Forgetting (Selective Promotion)
When the agent performs a `pushMemory` to create a summary node, it exercises **cognitive agency**:
- **Selective Referencing**: The agent chooses which `pointers` to include. Irrelevant chunks are omitted, effectively "forgetting" them from the global view while keeping them in the repository for deep-dive recovery.

## 5. Hybrid Memory Architecture: The Golden Fractal Graph

Memory is implemented as a **Directed Weighted Graph** using a Golden Ratio ($\phi \approx 1.618$) scaling system to ensure a natural proportion between summary and detail.

### A. The Golden Scale (Fractal Sizing)
To avoid arbitrary limits, memory blocks follow a fractal growth pattern based on the "Tweet" unit (280 bytes):

| Level | Name | Size (Approx) | Purpose |
| :--- | :--- | :--- | :--- |
| **L0** | **Teaser** | $\sim 280	ext{ bytes}$ | The "Twitter-sized" entry point. High-density summary. |
| **L1** | **Fragment** | $280^\phi \approx 6\text{--}9	ext{ KB}$ | The "Leaf". Detailed technical data, code snippets, or logic. |
| **L2** | **Context** | $\text{L1} \times \phi \approx 10\text{--}15	ext{ KB}$ | The "Branch". Synthesis of multiple fragments. |
| **L3** | **Root** | $\text{L2} \times \phi \approx 16\text{--}25	ext{ KB}$ | The "Trunk". Global project state and mission. |

### B. Cognitive Agency & Selective Purging
The agent is not a passive recorder but an active curator of its own memory. It must exercise **Cognitive Agency** during every `pushMemory` operation:

1. **Noise Filtering**: The agent must evaluate if a piece of information is "transient" (useful only for the current turn) or "persistent" (useful for future sessions). Transient data is **purged** and never written to the graph.
2. **Selective Promotion**: Only information that passes the relevance filter is promoted from the volatile context to a Fractal Node.
3. **Active Pruning**: During synthesis, the agent may decide to delete or merge old fractal nodes that are no longer accurate or have been superseded by new truths.

Memory is implemented as a **Directed Weighted Graph** where time and relationship are first-class citizens.

### A. The Semantic Network & Weighted Citations
- **Dynamic Graph**: Memory is a living network. A node is a hub of relationships with a calculated **Weight** based on its popularity (number of citations).
- **Automatic Back-Linking**: When a node is cited, the tool updates the cited node's footer and increments its weight. This ensures a node knows exactly who references it without redundant entries.
- **Memory Citations (Footnotes)**: The agent uses the format `[#ID, YYYY-MM-DD HH:MM]` to cite previous memories.
- **Node Structure**: Every memory block follows a strict layout:
    1. **Header**: Role summary (< 80 chars).
    2. **Body**: The content (with citations embedded).
    3. **Footer**: "Referenced by:" list using the citation format `[#ID, Date]`.
- **Atomic Chunks (The Leaves)**: Detailed information in blocks of **1KB to 2KB**.
- **The "Twitter" Rule (The Nodes)**: Every node has a summary limited to $\sim$280 characters.

### B. The Temporal Dimension (The Chronos)
- **System-Managed Time**: Timestamps are handled exclusively by the tool. The agent **sees** the dates in the `pullMemory` results to maintain temporal awareness, but **never writes** them.
- **The Narrative Thread**: The "story" is a linear traversal of the graph sorted by timestamp.
- **The Horizon (Look-Ahead)**: Navigating the thread returns the **Teaser + Timestamp** of the next/previous node. This allows the agent to perceive the distance in time between memories before deciding to anchor.

### C. Navigation Modes

Depending on the goal, the agent should use different navigation modes:

| Mode | Goal | Tool Sequence |
| :--- | :--- | :--- |
| **Analytical** | Understand a specific logic/feature | `pullMemory(key=X)` $\rightarrow$ `direction="down"` (details) $\rightarrow$ `direction="up"` (context). |
| **Narrative (The ID Chain)** | Trace the logical evolution of the project | `pullMemory(direction="prev")` $\rightarrow$ `pullMemory(direction="prev")`. This follows the **Sequential ID Chain**, allowing the agent to rewind the project's history linearly. |
| **Discovery** | Find related concepts | `pullMemory(key=X)` $\rightarrow$ Check "Referenced by" footer $\rightarrow$ Jump to cited IDs. |

**The ID Chain Principle**: Since IDs are auto-incrementing and timestamped, the sequence of IDs represents the **Chronological Truth** of the project. The agent can use this chain to reconstruct the narrative of how a decision was reached, even if the semantic links (edges) are sparse.

## 6. Technical Database Schema (SQLite Implementation)

### A. Table: `nodes`
- `id` (INTEGER PRIMARY KEY)
- `timestamp` (INTEGER): UTC Unix Epoch.
- `role` (TEXT): The role of the agent when this memory was created.
- `type` (TEXT): 'tattoo', 'snapshot', 'fractal'.
- `content` (TEXT): Full data (Header: Role summary < 80 chars | Body: Content with citations | Footer: Referenced-by list).
- `teaser` (TEXT): The "Twitter-sized" summary ($\sim$280 chars).
- `weight` (INTEGER): Number of incoming citations.
- `is_vital` (BOOLEAN): If true, loaded during the "Mirror" effect.

### B. Table: `edges`
- `source_id` (INTEGER)
- `target_id` (INTEGER)
- `relation_type` (TEXT): 'parent', 'child', 'associative'.

### C. Query Logic
- **`pullMemory(direction="next")`**: `SELECT teaser FROM nodes WHERE timestamp > (current_timestamp) ORDER BY timestamp ASC LIMIT 1`.
- **`pullMemory(direction="down")`**: `SELECT teaser, id FROM nodes JOIN edges ON nodes.id = edges.target_id WHERE edges.source_id = current_id AND edges.relation_type = 'child'`.
- **`pushMemory(pointers=[...])`**: Creates a new node and multiple entries in `edges` linking the new node to the provided IDs.

## 7. Cognitive Safety & Risk Mitigation

### A. The Verification Loop (Anti-Hallucination)
- **Cross-Reference**: Before using `pushMemory` for a 'tattoo' or 'fractal' node, the agent must verify the information against the current codebase or user confirmation.
- **Conflict Flagging**: If a new fact contradicts an existing tattoo, the agent must flag a `CRITICAL_INCONSISTENCY` and request user intervention.

### B. Circuit Breakers (Anti-Looping)
To prevent "Infinite Introspection" (cognitive loops where the agent cycles through memories without acting), the following safeguards are implemented:

1. **Loop Detection**: The `pullMemory` tool tracks the sequence of visited IDs. If a cycle is detected (A $\rightarrow$ B $\rightarrow$ A), the tool returns the content with a **Loop Warning**, prompting the agent to return to the Root Summary.
2. **The Mission Anchor**: The agent is forbidden from performing more than 5 consecutive `pullMemory` calls without re-validating its current path against the Root Summary/Mission.
3. **Exploration Budget**: If the agent spends too many tokens in memory navigation without producing an external output (code change or user response), it must trigger a "Context Reset". This reset forces the agent to execute `pullMemory()` (The Mirror Effect) to re-anchor itself to the Protocol and the Root Mission.
